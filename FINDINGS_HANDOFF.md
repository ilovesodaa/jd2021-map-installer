# Investigation Handoff: Audio Preview & Audio/Video Sync

**Date:** 2026-02-22  
**Project:** Just Dance Unlimited → JD2021 PC Dev Build map conversion (case study: Starships by Nicki Minaj)  
**Context:** This document summarises completed investigation findings for the next agent/session to act on.

---

## Finding 1 — Audio Preview Logic (RESOLVED, no code change yet)

### Summary
The engine does **not** use a separate preview audio file. It uses beat-based seek values in the `.trk` file to play a portion of the **main audio** as the preview. The `Starships_AudioPreview.ogg/wav` files that `restore_starships_media.py` copies and converts are **completely orphaned** — no actor, ISC, TPL, or scene file references them at any point in the build pipeline.

### Evidence
- `Starships_musictrack.tpl` (generated by `build_starships_fix.py`) references only `Starships.wav` (local) and `jmcs://jd-contents/Starships/Starships.ogg` (URL). No preview audio path.
- `Starships_audio.isc` contains exactly one audio actor: `MusicTrackComponent` → `Starships_musictrack.tpl`. No secondary preview actor.
- Reference map **GetGetDown** has **no `AudioPreview` file at all** — only `GetGetDown.wav`. It confirms the engine pattern: main audio + seek values = preview.
- The `.trk` `previewEntry`/`previewLoopStart`/`previewLoopEnd` fields serve as **both audio and video** preview seek points simultaneously.

### Current `.trk` preview values (Starships)
```
previewEntry      = 84.0   (beats)
previewLoopStart  = 84.0   (beats)
previewLoopEnd    = 244.0  (beats)
```

### Required Code Change
In `restore_starships_media.py`, remove (or skip) the copy and `ffmpeg` conversion of `Starships_AudioPreview.ogg` → `Starships_AudioPreview.wav`. The file does nothing and only wastes disk space and build time. When generalising for other songs, do not include a preview audio copy step.

## Finding 2 — Audio/Video Sync Issue (ROOT CAUSE IDENTIFIED, fix not implemented)

### Summary
Every timing value in the generated `Starships.trk` is a **verbatim copy** of the JDU source (`starships_musictrack.tpl.ckd`). There are zero transcription errors. The sync issue is not a script bug — it is a **platform interpretation difference** between the JDU smart TV platform (where the video was encoded) and JD2021 PC.

### Key Timing Values (from JDU source CKD)
| Field | Starships | GetGetDown (reference) |
|---|---|---|
| `videoStartTime` | **-1.901000 s** | -3.780000 s |
| `startBeat` | -4 | -6 |
| BPM | 125.0 | ~126.96 |
| Marker spacing | 23040 samples @ 48kHz | 22677 samples @ 48kHz |
| `previewEntry` | 84.0 beats | 51.0 beats |
| `previewLoopStart` | 84.0 beats | 48.0 beats |
| `previewLoopEnd` | 244.0 beats | 128.0 beats |

### Root Cause Hypothesis
At 125 BPM, `startBeat = -4` means audio pre-rolls by `4 × 0.48s = 1.920s`. But `videoStartTime = -1.901s` — so by design, audio leads video by ~19ms at the start.

GetGetDown's pre-roll is much longer (-3.780s), which gives the video more buffer time before the first beat. Starships' very short `-1.901s` pre-roll likely means the JD2021 PC engine doesn't have enough time to fully buffer/start the video before beat 0 arrives, causing drift.

Additionally: the JDU video file's actual visual beat 0 frame may have been calibrated for JDU's specific video stack (DASH/WebM on smart TV), and may not land at the exact millisecond the PC engine expects based on `-1.901s`.

### Secondary Finding: `.trk` format difference
Starships `.trk` sections use **integer marker values** and **no `comment` fields**:
```lua
{ MusicSection = { sectionType = 0, marker = 0 } }
```
GetGetDown uses **float markers** and **`comment = ""`**:
```lua
{ MusicSection = { sectionType = 0, marker = 0.000000, comment = "" } }
```
This is a cosmetic Lua difference (the engine parses both), but worth harmonising when generalising the build script.

### Required Code Change
In `build_starships_fix.py`, add a `videoStartTimeOverride` parameter (defaulting to `None`). When set, it replaces the `videoStartTime` value read from the JDU source CKD before writing the `.trk`. This allows per-song empirical correction without modifying source data.

**Steps to determine the correct override value:**
1. Open the Starships video file in a video editor or use `ffprobe` to find the exact timestamp of the first visible beat-0 cue (e.g., first coach move, first note hit).
2. The engine expects beat 0 to occur at timestamp `abs(videoStartTime)` from the start of the video file.
3. If beat 0 is visually at e.g. `1.950s` in the video but `videoStartTime = -1.901`, the override should be `-1.950`.
4. Adjust until in-game audio and video beat 0 align.
